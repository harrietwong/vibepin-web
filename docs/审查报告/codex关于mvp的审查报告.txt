# MVP 上线开发计划

## 总原则

1. 不重写 Create Pins、Plan 页面。
2. 不实施完整 Admin 后台。
3. 不扩展 Instagram、TikTok、Facebook。
4. 不把 Product Opportunity 重构混入 Create Pins/Plan 修复。
5. 每个工作包单独 Commit，便于最终审查和回滚。
6. 先冻结当前工作树，确保没有多人同时修改同一批文件。
7. 所有新增数据库迁移只编写文件，不擅自执行生产数据库。
8. Pinterest、Paddle 未获批期间使用 Sandbox，但代码路径必须与生产一致。

## WP1：统一 Schedule 和 Publish 校验

优先级：P0。

目标：防止非法 Draft 进入 Scheduled，统一 Create Pins、Plan、Batch Edit 的校验逻辑。

开发内容：

1. 建立共享 `validatePinForSchedule`。
2. 建立共享 `validatePinForPublish`。
3. Create Pins、Plan、Batch Edit、自动发布复用相同规则。
4. Schedule 时校验公开图片 URL。
5. Schedule 时校验 Pinterest Board。
6. Schedule 时校验 Pinterest 连接和 token 状态。
7. 非空 Website URL 必须是合法 HTTP/HTTPS。
8. Title 最大 100 字符。
9. Description 最大 500 字符。
10. Alt Text 是否必填按当前产品裁决统一，不得代码和测试各用一套。
11. 字段错误显示在字段附近，Toast 只做汇总。
12. Schedule 失败不得修改 Draft 生命周期。
13. Publish 失败不得丢失用户刚编辑的字段。
14. Publish/Schedule 前必须读取 store 最新 Draft，不能使用旧 React closure 快照。
15. 重复点击只允许产生一次请求。

建议产品裁决：

- Pinterest API 硬必填：图片、Board。
- VibePin 内容质量必填：Title、Description。
- Alt Text 推荐但不阻塞发布。
- Website URL 可空，但非空必须合法。

重点文件：

- `web/src/lib/pinReadiness.ts`
- `web/src/lib/weeklyPlanStats.ts`
- `web/src/components/studio/StudioBoard.tsx`
- `web/src/components/studio/BatchEditDrawer.tsx`
- `web/src/components/plan/DraftDetailsDrawer.tsx`
- `web/src/lib/smartSchedule.ts`
- `web/src/lib/studio/pinLifecycle.ts`
- `web/src/lib/server/pinterest/validatePublish.ts`

必须增加的测试：

1. 非法 Website URL 无法 Schedule。
2. 非法 Website URL 无法 Publish。
3. Board 缺失无法 Schedule。
4. 非公开图片无法 Schedule。
5. 编辑后立即 Schedule 使用最新字段。
6. 编辑后立即 Publish 使用最新字段。
7. 双击 Publish 只有一次 API 请求。
8. Batch 中一项失败不影响其他项。
9. Schedule 校验失败后仍是 Unscheduled。
10. 自动发布使用相同校验规则。

## WP2：修复自动发布时区和基础设施

优先级：P0；如果 MVP 不承诺自动发布，可改为暂缓并删除相关宣传。

目标：用户设置几点，就按用户所在时区几点发布。

开发内容：

1. 将 IANA timezone 持久化到 Smart Schedule 配置。
2. Pin Draft 排期时保存 timezone 或直接保存规范化 UTC `scheduledAt`。
3. 服务端不得再把本地墙钟字符串直接当 UTC。
4. 处理夏令时切换。
5. 明确不存在时间、无效 timezone、过去时间的行为。
6. 应用前检查 v38、v42 migration 是否存在。
7. 配置 `CRON_SECRET`。
8. 部署每五分钟运行的 Cron。
9. 增加 Cron 健康状态和最近成功时间。
10. 增加失败数、发布数、跳过数日志。
11. 对进程崩溃后的重复发布风险增加保护。
12. 自动发布失败写入与手动发布相同的 failure 字段。
13. 自动发布成功清理 schedule 并写入 remote Pin 信息。
14. Pinterest Standard Access 未获得时，不应把权限不足记录成普通内容失败。

重点文件：

- `web/src/app/api/pin-drafts/promote.ts`
- `web/src/app/api/pin-drafts/route.ts`
- `web/src/app/api/cron/publish-due/route.ts`
- `web/src/app/api/cron/publish-due/publishDueLogic.ts`
- `web/src/lib/smartScheduleStore.ts`
- `web/src/lib/smartSchedule.ts`
- `backend/db/migrate_v42_scheduled_publish.sql`
- `docs/运维/自动发布-cron配置.md`

验收场景：

1. 洛杉矶上午 9 点按洛杉矶上午 9 点发布。
2. 纽约上午 9 点按纽约上午 9 点发布。
3. 夏令时前后时间正确。
4. 两个 Cron 同时执行不会重复发布。
5. 新鲜 claim 不会被第二个 worker 获取。
6. 过期 claim 可以恢复。
7. 已 Posted 的 Draft 永远不会再次发布。
8. 失败 Draft 不会每五分钟无限重试。
9. 手动 Publish 与 Cron 不会同时发布同一 Draft。
10. Cron 未配置时有明确健康告警。

## WP3：迁移 AI 生图到稳定后端

优先级：P0。

目标：生产环境不再依赖 Vercel Node Route 启动本地 Python 进程。

推荐方案：

`Next.js → FastAPI generation endpoint → durable worker/job → LinAPI → Supabase Storage → Pin Draft Sync`

开发内容：

1. Next.js `/api/generate` 不再直接 `spawn generator.py`。
2. FastAPI 接收完整多模态生成参数。
3. 支持 Product Images、References、Creative Direction、Model、Count、Ratio。
4. 创建持久化 generation job。
5. 返回 jobId，不阻塞请求等待完整生成。
6. 前端通过轮询、SSE 或现有同步机制获取结果。
7. 用户关闭 Drawer 后任务继续。
8. 页面刷新后能恢复 Generating 状态。
9. Worker 崩溃后任务能够进入 Failed 或重试。
10. 部分生成成功时保留成功结果。
11. 每个结果仍创建独立 Pin Draft。
12. 成功图片必须上传稳定公共 URL。
13. 图片分析、关键词、Quality Judge 在生成成功后异步启动。
14. 生成锁不能依赖单机临时目录。
15. 去掉 Next Build 的全仓库文件追踪警告。

重点文件：

- `web/src/app/api/generate/route.ts`
- `web/src/lib/studio/generateAiVersions.ts`
- `web/src/components/studio/StudioBoard.tsx`
- `web/src/components/studio/AiVersionDrawer.tsx`
- `api/app/main.py`
- `api/app/workers/task_worker.py`
- `api/app/services/image_gen.py`
- `backend/generator.py`

验收标准：

1. 连续生成 50 个任务无永久 Generating。
2. Drawer 关闭后生成继续。
3. 页面刷新后任务恢复。
4. 两张请求返回一张时显示一成功、一失败。
5. Provider 超时产生可重试失败。
6. Worker 重启不会遗失任务。
7. 同一请求不会重复建立结果卡。
8. 图片 URL 可被匿名请求。
9. Pinterest 能读取最终图片 URL。
10. Vercel 不需要 Python 环境。
