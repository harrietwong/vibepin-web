## 2. 可救性判定（只读抓取）

### 2.1 方法学与抽样口径（**不是全量，请按此口径解读**）

- **抽样**：798 行按 **domain 分层随机**抽取，实际探测 **363 行（45.5%）**，每个域名至少 1 行、大域名按比例配额。随机种子固定（可复现）。
- **请求参数**：浏览器 UA + 完整浏览器头；超时 10s；跟随重定向（终点重新判定）；单页 ≤500KB；**全局限速 ≤2 req/s**；只 GET，绝不登录/提交表单。
- **外推方式**：采用**分层（post-stratification）估计**——按各域名族在 798 行中的真实占比对各族可救率加权求和，而不是直接用样本粗率。样本相对总体略微低配了 Etsy，故加权值（17.8%）比样本粗率（22.3%）更保守也更准。
- **判定口径（关键）**：本任务的"可救"必须**同时**满足两条——(a) URL 仍是有效商品详情页；(b) **能抓到真实的（非 pinimg）商品图**。**只满足 (a) 不算可救**：因为 T10 的核心修复动作就是换掉假商品图，抓不到真图的行修不了。

> **一处实现坑（已修，如实记录）**：首轮探测把 51% 的行误判为 "not_product"。根因是请求头带了 `Accept-Encoding: br` 而环境未装 brotli，br 压缩响应被解码成乱码，导致 Shopify 页的 `og:type=product` / `og:image` 全部读不到。**已移除 br 并加了"响应无法解码 → 判 error 而非 not_product"的保护**，本报告所有数字均为修复后重跑所得。（这也是为什么最终 not_product 只剩 7 行。）

### 2.2 判定结果（n=363）

| 判定 | 数量 | 占比 | 含义 |
|---|---:|---:|---|
| **可救 recoverable** | **81** | 22.3% | 商品页有效 **且**抓到真实商品图（图在商家 CDN） |
| **不可救 — blocked（WAF 403）** | **267** | **73.6%** | 站点对任何合规轻量 GET 一律 403，**抓不到商品图** |
| **不可救 — dead** | 8 | 2.2% | 404 (1) / 已售出下架文案 (3) / 重定向到首页或分类页 (4) |
| **不可救 — not_product** | 7 | 1.9% | 页面不是商品详情页 |
| **error（网络失败）** | **0** | **0%** | —— |

- 可救的 81 行中：抓到**价格 81/81**、**标题 77/81**、商品图 81/81（均在商家 CDN，如 `*/cdn/shop/files/*`）。
- **blocked 全部是 HTTP 403**，其中 **Etsy 系 240 行、canva 14、depop 4、ko-fi 3、ebay 2**。

### 2.4 可救率结论

> **加权外推可救率 ≈ 17.8%（约 142 / 798 行；95% 区间 14.1%–21.6%，约 113–172 行）。**
> 样本粗率 22.3%。**两个口径都远低于任务卡"可救率 <50% 时暂停并重新评估"的停止阈值。**

**Etsy 的 403 是结构性的、不是实现问题**：单独针对 Etsy 复测（正常浏览器头、无 br），返回的是固定 776 字节的拦截页，**240/240 全拦，零例外**。T1 dry-run 报告（§4.2）已独立观察到同一现象（Etsy/eBay/Depop 对轻量 GET 一律 403）。要拿到 Etsy 商品图，只能走 Etsy 官方 API 或渲染型/代理型抓取——前者需申请与配额，后者直接违反任务书 §6 红线第 4 条"外部商品页验证为只读轻量请求"。**在合规边界内，本批 70.4% 的行的商品图不可能被重新抓到。**

## 4. 建议：**整批退役（Retire），不做修复**

### 4.1 判断依据

1. **可救率 17.8%，触发任务卡停止条件（<50%）。** 任务卡原文即写明"可能直接退役整批更划算"——数据支持这条。
2. **修复后剩下的东西也不值钱。** 即使把 ~142 行全修好，它们也只是 142 行商品——而 T2 已经证明，用干净管线**新采 100 行只需一次 bounded apply**。与其花大力气抢救 142 行历史行，不如直接用干净管线再跑几批，语料更新、字段更全（T2 批次 price/source_pin_saves/真商品图全齐，脏批 price 全空）。**这批数据的边际价值 ≈ 用新管线重采的成本，而后者显著更低、且零污染风险。**
3. **"修复"其实是"重建"。** 这批行需要重写的字段包括 image_url、save_count、source_pin_saves、source_pin_image_url、price、product_name（现存的是 Pin 标题不是商品标题）——**几乎每一个业务字段都要换**。唯一能保留的是 parent_pin_id / source_url。这已经不是清洗，是拿旧行当壳重新灌数据，风险与收益都不如 insert 新行。
4. **部分修复会制造更隐蔽的坑。** 若只修那 ~142 行、把其余 656 行留在库里，则同一 `discovery_method` 下会同时存在干净行与脏行，未来任何"按 discovery_method 整批回滚/统计"的操作都会踩到混合语义；而 §6 红线第 3 条要求"任何写入必须可按 discovery_method + 时间窗口整批回滚"。**要么整批干净，要么整批退役。**
5. **止血现状已经生效。** `/api/products/top` 的 `created_at >= 2026-07-13T14:28:03Z` 下限已经把这 798 行隔离为不可见，**线上此刻没有在展示脏数据**。所以这不是一个"必须马上抢救"的线上事故，而是一笔可以从容清掉的技术债。

### 4.2 推荐执行路径（**需决策人单独授权后**，本次不执行）

**方案 A（推荐）：整批退役 + 撤除临时下限**

1. 先导出 798 行全量原始值到 evidence 文件（任务卡 §写入/回滚已要求）；
2. **退役方式二选一，需决策人明确选择**（任务卡 c 项要求）：
   - **A1 软退役（更稳，推荐）**：加一列 `retired_at` / 或复用现有可见性字段，把 798 行标记为不可见。**可逆**，STL 零影响。
   - **A2 硬删除**：`DELETE FROM pin_products WHERE discovery_method='outbound_link_bootstrap'`（798 行，单条 SQL，不触碰任何 STL 行）。**不可逆**，须先有 evidence 导出。
3. 退役后，`/api/products/top` 的 `created_at` 临时下限即可**安全撤除**——因为脏行已不在可见集，下限失去存在意义（这正是任务卡 d 项的目标，且撤除后不再需要每来一批 outbound 就手调常量）；
4. 重跑 T3 六项验证 + 确认 STL 零回归；
5. 用干净的 T2 管线补量（这才是把 Product 池做厚的正路）。

**方案 B（不推荐）：混合修复**——修 ~142 行、退役 656 行。多一套按行 UPDATE 的代码与快照/回滚负担，却只换来 142 行商品，且留下同一 discovery_method 下语义混合的隐患。**除非决策人认为这 142 行有不可替代的价值，否则不建议。**

### 4.3 若仍决定修复：工作量与外部抓取量预估

- **外部抓取量**：需对**全部 798 行**做一次只读 GET 才能逐行定可救性（本次只抽了 363）；按 ≤2 req/s 限速，**798 次请求 ≈ 7–10 分钟**（本次 363 行实测无失败、无 429，抓取本身不是瓶颈）。
- **真正的成本不在抓取，在写入侧**：需要新写一个按行 UPDATE + 全量前置快照 + 可整批回滚的脚本（脏批没有干净的时间窗口可用作回滚键——798 行 created_at 横跨 6/01–7/09，与其他批次时间上不隔离，**只能靠 id 列表回滚**，比 T2 的"时间窗口一条 SQL 回滚"脆弱得多）。
- **产出**：约 **142 行**可修（其中 Etsy 0 行、Shopify 独立站占绝大多数），**656 行仍需退役**。
- **一句话**：**花 A 方案数倍的工程量与回滚风险，换 142 行本可以用干净管线低成本重采的商品。不划算。**
